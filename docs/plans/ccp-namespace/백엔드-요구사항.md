# CCP Namespace 백엔드 요구사항

## 목적
- `apps/web/src/content/ccp/cicd/namespace/` 화면이 실제 동작하려면 필요한 백엔드 기능을 정리한다.
- 이 문서는 UI 목업 기준의 기능 목록이며, 이후 API 명세와 작업 분해의 기준으로 사용한다.

## 전제
- 이 기능은 단순 DB CRUD가 아니라 두 Git 레포를 함께 변경하는 오케스트레이션이다.
- 대상 레포:
  - `k8s-cicd-init`
  - `k8s-app-init`
- 주요 동작:
  - 템플릿 복제
  - 템플릿 문자열 치환 (`sample -> {ns}`)
  - `kustomization.yaml` 참조 추가/삭제
  - `clusterrolebinding.yaml` 갱신

## 템플릿 복사 기준

### 1. `k8s-cicd-init`
- 복사 기준 폴더:
  - `sample-cicd/` -> `{ns}-cicd/`
- 함께 수정해야 하는 공용 파일:
  - 루트 `kustomization.yaml`의 `resources`에 `{ns}-cicd` 추가/삭제
  - `clusterrolebinding.yaml`의 `subjects`에 ServiceAccount 항목 추가/삭제
- `clusterrolebinding.yaml` 처리 방식:
  - 새 문서를 별도로 추가하는 방식이 아니라, 기존 ClusterRoleBinding의 `subjects` 배열에 계속 추가하는 방식으로 처리한다.

### 2. `k8s-app-init`
- 환경별 복사 기준 폴더:
  - `overlays/dev/sample-dev/` -> `overlays/dev/{ns}-dev/`
  - `overlays/stg/sample-stg/` -> `overlays/stg/{ns}-stg/`
  - `overlays/prd/sample-prd/` -> `overlays/prd/{ns}-prd/`
- 함께 수정해야 하는 공용 파일:
  - 각 환경의 `overlays/{env}/kustomization.yaml`에 `{ns}-{env}` 추가/삭제
- 보안 정책 처리:
  - `dev`의 `sample-dev/kustomization.yaml`이 참조하는 `../../_security/sample`도 함께 복사해야 한다.
  - 즉 `_security/sample/` -> `_security/{ns}/` 복사가 필요하다.
- VirtualService 처리:
  - `overlays/{env}/istio-system/sample-vs.yaml` 계열은 이번 범위에서 생성하지 않는다.

### 3. 커스텀 환경 생성 규칙
- `qa`, `perf`, `canary` 같은 커스텀 환경도 내부 내용은 기존 sample 템플릿과 동일하게 생성한다.
- 즉 새 환경도 기본적으로 `sample` 템플릿을 기준으로 만들고, 이름과 환경 suffix만 달라진다.
- 백엔드는 새 환경 생성 시 아래를 처리해야 한다:
  - `overlays/{new-env}/` 구조 준비
  - `sample-{new-env}`에 해당하는 새 namespace overlay 생성
  - 해당 환경의 `kustomization.yaml` 참조 반영

### 4. 예외 및 보정 규칙
- `overlays/stg/sample-stg/kustomization.yaml`에는 현재 샘플 값 오류가 있다.
- 복사 시 이 값을 그대로 따르지 말고 정상적으로 `{ns}-stg`로 교정해야 한다.

## 화면별 요구사항

### 1. Namespace 목록 화면
- 대상 화면: `CCP-NS-001`
- 필요한 기능:
  - 목록 조회
  - 이름 검색
  - 환경 필터
- 응답에 필요한 필드:
  - NS 이름
  - 연결된 배포 환경 목록
  - 생성일

#### 목록 데이터 산출 규칙
- 목록의 기준 소스는 **실제 Kubernetes Namespace** 이다.
- 조회 대상은 Namespace 라벨 중 아래 조건을 만족하는 것만 포함한다:
  - `cone-chain-portal.managed-by=CI_CD`
- 목록에 보여주는 이름은 실제 Namespace 이름에서 뒤의 `-cicd`를 제거한 값이다.
  - 예: `sample-cicd` -> `sample`
  - 예: `payment-cicd` -> `payment`
- 즉, 목록의 기준 Namespace는 사실상 `{ns}-cicd` 형식의 CI/CD Namespace 이다.

#### 배포 환경 계산 규칙
- 배포 환경은 별도 필드로 저장된 값을 쓰는 것이 아니라, **같은 prefix를 가진 Namespace/리소스 이름을 기준으로 유도**한다.
- 목록에 표시할 NS 이름이 `sample`이면, `sample-`으로 시작하는 이름들 중 뒤 suffix를 환경명으로 해석한다.
  - 예: `sample-dev` -> `dev`
  - 예: `sample-stg` -> `stg`
  - 예: `sample-prd` -> `prd`
- 즉, `-` 뒤의 값을 배포 환경으로 사용한다.
- 화면에는 이 값을 `dev`, `stg`, `prd` 같은 뱃지로 표시한다.

#### 생성일 계산 규칙
- 생성일은 목록 기준 Namespace 자체에서 바로 쓰는 것이 아니라, **cicd 쪽 라벨 정보에서 찾아야 한다**.
- 즉 `{ns}-cicd`에 달린 라벨 또는 메타데이터에서 생성일 값을 읽어 목록의 `createdAt`으로 내려준다.
- 백엔드는 생성일 라벨 키를 명확히 정의해야 한다.
  - 예: `cone-chain-portal.created-at`
  - 또는 기존에 이미 쓰는 별도 라벨 키
- 프론트에는 최종적으로 표시 가능한 날짜 문자열 또는 ISO datetime을 내려준다.

#### 목록 조회 응답 예시 개념
- `sample-cicd` Namespace 발견
- `cone-chain-portal.managed-by=CI_CD` 확인
- 목록 이름은 `sample`
- 같은 prefix의 환경 리소스에서 `sample-dev`, `sample-stg`, `sample-prd`를 찾아 환경 목록 구성
- `{ns}-cicd` 라벨에서 생성일을 읽어 `createdAt` 구성

### 2. Namespace 생성 화면
- 대상 화면: `CCP-NS-002`
- 필요한 기능:
  - 이름 중복 검사
  - 생성 가능 환경 목록 조회
  - Namespace 생성
- 생성 시 서버가 처리해야 하는 작업:
  - `k8s-cicd-init`에 `{ns}-cicd/` 생성
  - 내부 템플릿 문자열 치환
  - 루트 `kustomization.yaml` 참조 추가
  - `clusterrolebinding.yaml`의 `subjects`에 ServiceAccount 권한 추가
  - 선택한 환경별로 `k8s-app-init`에 `overlays/{env}/{ns}-{env}/` 생성
  - 각 환경 `kustomization.yaml` 참조 추가
  - 필요한 경우 `_security/sample` -> `_security/{ns}` 복사

### 3. Namespace 상세 요약 화면
- 대상 화면: `CCP-NS-004`
- 필요한 기능:
  - Namespace 상세 조회
- 응답에 필요한 필드:
  - NS 이름
  - 상태
  - `k8s-cicd-init/{ns}-cicd/` 경로
  - 연결된 환경 목록
  - Labels
  - 생성일

#### 상세조회 요청 기준
- 프론트는 화면에 보이는 NS 이름 `{ns}` 로 상세조회 요청을 보낸다.
- 예:
  - 요청값: `sample`
  - 실제 기준 Namespace: `sample-cicd`

#### `CCP-NS-004` 기준 상세 조회 필수 정보
- 기본 정보 카드:
  - `name`
  - `status`
  - `cicdRepoPath`
  - `labels[]`
  - `createdAt`
- 배포 환경 영역:
  - `environmentCount`
  - `environments[]`
- 환경 테이블의 각 행:
  - `envName`
  - `status`
  - `appInitPath`

#### `CCP-NS-004`에서 실제로 보이는 값
- 상세 패널 상단 title:
  - 화면용 NS 이름 `{ns}`
- 기본 정보:
  - 이름: `sample`
  - 상태: `Active`
  - CICD 레포: `k8s-cicd-init/sample-cicd/`
  - Labels: 최소 `cone-chain-portal.managed-by: CI_CD`
  - 생성일: `sample-cicd` 기준 생성일
- 배포 환경:
  - 환경 개수 뱃지
  - 각 환경 이름: `dev`, `stg`, `prd`
  - 각 환경 상태
  - 각 환경 경로: `overlays/{env}/{ns}-{env}/`

#### 상세 조회 데이터 산출 규칙

##### 1. 기준 Namespace 조회
- 먼저 Kubernetes Namespace 목록에서 이름이 `{ns}-cicd` 인 Namespace를 찾는다.
- 이 Namespace가 상세조회의 기준 원본이다.
- 아래 조건이 맞아야 유효한 상세조회 대상이다:
  - Namespace 이름: `{ns}-cicd`
  - 라벨: `cone-chain-portal.managed-by=CI_CD`

##### 2. 기본 정보 카드 구성 규칙
- `name`
  - `{ns}-cicd` 에서 `-cicd`를 제거한 화면용 이름을 사용한다.
  - 예: `sample-cicd` -> `sample`
- `status`
  - `{ns}-cicd` Namespace의 실제 Kubernetes 상태를 사용한다.
  - 화면에는 `Active` 같은 표시값으로 변환해 내려준다.
- `cicdRepoPath`
  - 고정 규칙으로 계산한다.
  - 값: `k8s-cicd-init/{ns}-cicd/`
- `labels`
  - `{ns}-cicd` Namespace의 라벨 목록을 key/value 형태로 내려준다.
  - 최소 포함 대상:
    - `cone-chain-portal.managed-by=CI_CD`
- `createdAt`
  - `{ns}-cicd` 기준 생성일을 사용한다.
  - 이전 규칙대로 cicd 쪽 라벨/메타데이터에서 읽는다.

##### 3. 배포 환경 목록 계산 규칙
- 배포 환경은 별도 저장값이 아니라 같은 prefix를 가진 Namespace들로부터 계산한다.
- 기준은 아래와 같다:
  - 이름이 `{ns}-` 로 시작한다.
  - `{ns}-cicd` 는 제외한다.
  - 마지막 `-` 뒤 suffix를 환경명으로 사용한다.
- 예:
  - `sample-dev` -> `dev`
  - `sample-stg` -> `stg`
  - `sample-prd` -> `prd`

##### 4. 환경 테이블 row 구성 규칙
- `envName`
  - 환경 Namespace 이름의 suffix 값
- `status`
  - 해당 환경 Namespace의 실제 상태를 사용한다.
  - 화면용 값은 `Active` 같은 문자열로 변환한다.
- `appInitPath`
  - 고정 규칙으로 계산한다.
  - 값: `overlays/{env}/{ns}-{env}/`
  - 예: `overlays/dev/sample-dev/`

##### 5. 환경 개수 계산 규칙
- `environmentCount`는 계산된 `environments[]` 길이와 동일하다.
- 예:
  - `sample-dev`, `sample-stg`, `sample-prd`가 있으면 `3`

#### 상세 조회 처리 순서
1. 요청으로 받은 `{ns}`를 기준으로 `{ns}-cicd` Namespace를 찾는다.
2. `cone-chain-portal.managed-by=CI_CD` 라벨을 검증한다.
3. `{ns}-cicd`에서 기본 정보 카드 데이터를 만든다.
4. 같은 prefix `{ns}-`를 가진 환경 Namespace를 수집한다.
5. `-cicd`는 제외하고 suffix를 환경명으로 변환한다.
6. 각 환경의 상태와 `appInitPath`를 계산한다.
7. `environmentCount`와 `environments[]`를 조립해 응답한다.

#### 예시: `sample` 상세조회
- 요청값: `sample`
- 기준 Namespace:
  - `sample-cicd`
- 기본 정보:
  - `name = sample`
  - `status = Active`
  - `cicdRepoPath = k8s-cicd-init/sample-cicd/`
  - `labels = sample-cicd Namespace labels`
  - `createdAt = sample-cicd 기준 생성일`
- 환경 목록:
  - `sample-dev`
  - `sample-stg`
  - `sample-prd`
- 최종 화면 표시:
  - `dev`
  - `stg`
  - `prd`

## 구현 시 주의사항
- 생성/삭제는 요청 즉시 끝나지 않을 수 있으므로 비동기 작업 기반 처리가 적절하다.
- 두 레포 중 하나만 반영되는 부분 성공 상태를 추적할 수 있어야 한다.
- NS 이름 규칙과 환경 이름 규칙은 서버에서 최종 검증해야 한다.
- 현재 UI 기준으로 이름 변경 기능은 필요 없다.
- 현재 UI 기준으로 YAML 원문 조회 기능은 우선순위가 낮다.
- VirtualService는 현재 범위에서 제외한다.
- `stg` 샘플의 잘못된 namespace 값은 복사 로직에서 보정해야 한다.
