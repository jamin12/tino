# AI 작업 기억 시스템

AI는 대화가 길어지면 자신이 무엇을 하고 있었는지 잊어버린다.
이 시스템은 **3가지 필수 문서**를 AI의 외부 기억 장치로 활용하여 이 문제를 해결한다.
새로운 대화창에서도 문서를 읽으면 이전 맥락을 완전히 복구할 수 있다.

## 적용 시점

- `plan` 스킬 실행 시 자동으로 이 시스템 적용
- 복잡한 작업(3단계 이상)을 시작할 때
- 사용자가 "계획", "plan" 키워드 사용 시
- 장기 작업으로 컨텍스트 유실 위험이 있을 때

---

## 규칙 목록

### CRITICAL — 3종 문서 (외부 기억 장치)

#### `mem-three-docs`

모든 계획은 반드시 3개의 문서를 생성한다. 이 문서들은 AI의 외부 기억 장치 역할을 한다.

```
docs/plans/{plan-name}/
├── 계획서.md        # 설계도: 무엇을 만들지 처음부터 끝까지 정리
├── 맥락노트.md      # 시방서: 왜 이렇게 결정했는지 이유와 관련 자료 위치
└── 체크리스트.md     # 공정표: 끝낸 작업과 남은 작업을 추적
```

- `{plan-name}`은 작업 내용을 나타내는 kebab-case 이름 (예: `chart-alert-api`, `auth-refactor`)
- 세 문서 중 하나라도 빠지면 안 된다

#### `mem-plan-first`

무작정 코딩을 시작하지 않는다. 반드시 아래 순서를 따른다:

```
# WRONG — 문서 없이 바로 코딩 시작
사용자: "알림 API 만들어줘"
AI: (바로 코드 작성 시작)

# CORRECT — 계획 수립 → 검토 → 문서 저장 → 작업
사용자: "알림 API 만들어줘"
AI: 1) 코드베이스 탐색
    2) 계획 수립 및 3종 문서 작성
    3) 사용자에게 꼼꼼한 검토 요청
    4) 사용자 승인 후 작업 시작
```

#### `mem-small-batch`

한 번에 모든 작업을 하지 않는다. **한두 개씩** 진행하고 체크리스트를 업데이트한다.
이렇게 해야 AI가 길을 잃지 않는다.

```
# WRONG — 모든 작업을 한꺼번에
AI: (10개 파일을 동시에 수정하고 한참 뒤에 결과 보고)

# CORRECT — 1~2개씩 진행, 매번 체크리스트 업데이트
AI: 작업 1 완료 → 체크리스트.md 업데이트 → 작업 2 진행 → 체크리스트.md 업데이트 → ...
```

#### `mem-checklist-update`

작업 중 체크리스트를 **실시간으로 업데이트**한다. 각 단계를 시작할 때 `[진행중]`, 완료하면 `[완료]`로 변경한다.

```markdown
# WRONG — 체크리스트를 한 번 쓰고 방치
- [ ] API 엔드포인트 구현
- [ ] 서비스 레이어 구현
(작업 끝날 때까지 업데이트 안 함)

# CORRECT — 단계마다 파일에 직접 상태 업데이트
- [x] API 엔드포인트 구현 [완료]
- [>] 서비스 레이어 구현 [진행중]
- [ ] 테스트 작성 [대기]
```

### HIGH — 각 문서 작성법

#### `mem-doc-plan`

**계획서.md** (설계도) — 무엇을 만들지 처음부터 끝까지 정리한다.
새 대화에서 이 파일만 읽어도 "무엇을 해야 하는지" 완전히 이해할 수 있어야 한다.

```markdown
# {작업 제목}

## 목표
- 이 작업이 달성하려는 것을 명확하게 기술

## 배경
- 왜 이 작업이 필요한지, 현재 어떤 문제가 있는지

## 접근 방식
- 기술적 접근 방법을 구체적으로 서술
- 선택한 방식의 근거

## 기술적 결정 사항
| 결정 | 선택 | 이유 |
|------|------|------|
| 예: DB 스키마 | PostgreSQL jsonb | 유연한 스키마 필요 |

## 영향 범위
- 변경되는 파일/모듈 목록
- 기존 기능에 대한 영향

## 예상 결과
- 작업 완료 후 기대되는 상태
```

#### `mem-doc-context`

**맥락노트.md** (시방서) — 왜 이렇게 결정했는지 이유와 관련 자료 위치를 정리한다.
새 대화에서 이 파일을 읽으면 "왜 이런 방식으로 하는지" 즉시 파악할 수 있어야 한다.

```markdown
# 맥락 노트: {작업 제목}

## 현재 코드베이스 상태
- 관련 모듈의 현재 구조와 상태 요약

## 핵심 파일
| 파일 경로 | 역할 | 변경 여부 |
|-----------|------|-----------|
| src/... | ... | 신규/수정/참조 |

## 제약 조건
- 기술적 제약 (아키텍처, 의존성 등)
- 비즈니스 제약 (일정, 요구사항 등)

## 핵심 결정 로그
| 시점 | 결정 | 이유 | 대안 |
|------|------|------|------|
| 초기 | ... | ... | ... |

## 주의사항
- 작업 중 반드시 기억해야 할 사항
- 함정이나 실수하기 쉬운 부분

## 참고 자료
- 관련 문서, API 문서, 이슈 링크 등
```

#### `mem-doc-checklist`

**체크리스트.md** (공정표) — 끝낸 작업과 남은 작업을 추적한다.
새 대화에서 이 파일을 읽으면 "지금 어디까지 했고, 뭐가 남았는지" 바로 알 수 있어야 한다.

```markdown
# 체크리스트: {작업 제목}

## 진행 상태: {N}/{Total} 완료

## 작업 목록

### Phase 1: {단계명}
- [x] 작업 1 [완료]
- [>] 작업 2 [진행중]

### Phase 2: {단계명}
- [ ] 작업 3 [대기]
- [ ] 작업 4 [대기]

## 완료 기준
- [ ] 빌드 성공
- [ ] 테스트 통과
- [ ] 기능 동작 확인

## 이슈 & 블로커
| # | 이슈 | 상태 | 해결 방법 |
|---|------|------|-----------|
```

### HIGH — 워크플로우

#### `mem-workflow`

AI 작업 기억 시스템의 전체 워크플로우:

```
[같은 대화 내]
1. [탐색]  코드베이스 탐색으로 현재 상태 파악
2. [문서]  3종 문서 작성 (계획서 → 맥락노트 → 체크리스트)
3. [검토]  사용자에게 문서를 꼼꼼히 검토해달라고 요청
4. [승인]  사용자 승인 후 작업 시작
5. [실행]  체크리스트에서 1~2개씩 작업 수행
   - 각 단계 시작 시: 체크리스트.md [진행중] 업데이트
   - 각 단계 완료 시: 체크리스트.md [완료] 업데이트
   - 중요 결정 발생 시: 맥락노트.md 업데이트
6. [반복]  5번을 체크리스트가 끝날 때까지 반복

[새 대화창에서 이어갈 때]
1. docs/plans/{plan-name}/체크리스트.md → 어디까지 했는지 파악
2. docs/plans/{plan-name}/맥락노트.md → 핵심 결정과 제약 확인
3. docs/plans/{plan-name}/계획서.md → 전체 목표 확인
4. 체크리스트의 남은 작업부터 이어서 진행
```

#### `mem-new-conversation`

새 대화창에서 작업을 이어가야 할 때, 사용자가 plan-name을 알려주면
3종 문서를 읽고 맥락을 완전히 복구한 뒤 남은 작업을 이어간다.

```
# 새 대화에서 재개하는 방법
사용자: "docs/plans/chart-alert-api 이어서 해줘"
AI: 1) 체크리스트.md 읽기 → 현재 진행 상태 파악
    2) 맥락노트.md 읽기 → 결정 사항과 제약 확인
    3) 계획서.md 읽기 → 전체 목표 확인
    4) "N/Total 완료 상태입니다. 다음 작업은 X입니다." 보고
    5) 사용자 확인 후 이어서 진행
```

### MEDIUM — 문서 관리

#### `mem-self-contained`

3종 문서는 **자기 완결적**이어야 한다. 이전 대화 내용을 몰라도 문서만으로 모든 맥락을 파악할 수 있어야 한다.

```
# WRONG — 대화에 의존하는 내용
맥락노트: "위에서 논의한 대로 진행"
체크리스트: "앞서 말한 방식으로 구현"

# CORRECT — 자기 완결적 내용
맥락노트: "헥사고날 아키텍처에 따라 InPort/OutPort 분리. AlertService는 SaveAlertOutPort에만 의존."
체크리스트: "AlertController REST API 구현 (POST /api/alerts, GET /api/alerts/{id})"
```

#### `mem-naming`

plan-name은 작업 내용을 명확히 나타내는 kebab-case를 사용한다.

```
# WRONG
docs/plans/plan1/
docs/plans/새작업/
docs/plans/2026-02-25/

# CORRECT
docs/plans/chart-alert-api/
docs/plans/auth-refactor/
docs/plans/trading-phase-detection/
```

#### `mem-archive`

작업이 완전히 완료되면 체크리스트 상단에 완료 표시를 추가한다. 문서는 삭제하지 않고 이력으로 보존한다.

```markdown
# 체크리스트: 차트 알림 API
## 상태: 완료 (2026-02-25)
## 진행 상태: 8/8 완료
```
