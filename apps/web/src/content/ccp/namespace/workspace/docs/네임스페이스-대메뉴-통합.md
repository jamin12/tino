# 네임스페이스 대메뉴 통합 기획

> Dooray: CONE-Chain-Portal/367
> 작성일: 2026-03-18

---

## 1. 배경

### 현재 구조

CCP 대시보드 사이드 메뉴에서 "네임스페이스"는 **대메뉴로 존재하지만 하위 기능이 없는 빈 메뉴**이다.
실제 네임스페이스 관련 기능(생성/상세/삭제/환경추가)은 **CI/CD 하위 메뉴**에 위치해 있다.

```
현재 사이드 메뉴:
├─ 대시보드
├─ 네임스페이스        ← 대메뉴 (빈 메뉴)
├─ 애플리케이션
├─ CI/CD
│   ├─ 네임스페이스    ← 실제 기능은 여기
│   ├─ 저장소
│   ├─ 파이프라인
│   └─ 카탈로그
├─ GitOps
└─ ...
```

### 문제점

1. **메뉴 중복**: "네임스페이스"가 대메뉴와 CI/CD 하위에 동시에 존재하여 사용자 혼란 유발
2. **역할 불일치**: 네임스페이스 생성은 단순히 K8s namespace 하나를 만드는 것이 아님. `k8s-cicd-init` + `k8s-app-init` 두 Gitea 저장소에 아래 리소스를 동시에 프로비저닝하는 행위:
   - Namespace 정의
   - RBAC (ServiceAccount, ClusterRoleBinding)
   - PV / PVC (local + store)
   - LimitRange, LocalQueue (Kueue)
   - Secret (Git, Docker, Nexus, Kubeconfig, SSH, TLS, ArgoCD 등)
   - Istio Gateway + VirtualService
   - Istio 보안 정책 (RequestAuthentication, AuthorizationPolicy)
   → CI/CD의 하위 기능이라기보다 **독립적인 대메뉴 수준의 기능**에 해당
3. **기능 부재**: K8s namespace를 단일 리소스로 직접 생성/삭제하는 기능이 없음 (kubectl 대체 필요)

## 2. 변경 방향

CI/CD 하위의 네임스페이스 기능을 **대메뉴 "네임스페이스"로 편입**하고, 하위에 두 가지 메뉴를 구성한다.

```
변경 후 사이드 메뉴:
├─ 대시보드
├─ 네임스페이스
│   ├─ 워크스페이스          ← 묶음 프로비저닝 (기존 기능 이동)
│   └─ 네임스페이스 리소스    ← 신규 (K8s namespace 단일 CRUD)
├─ 애플리케이션
├─ CI/CD
│   ├─ 저장소               ← 네임스페이스 제거됨
│   ├─ 파이프라인
│   └─ 카탈로그
├─ GitOps
└─ ...
```

### 하위 메뉴 정의

| 메뉴 | 역할 | 설명 |
|------|------|------|
| **워크스페이스** | 묶음 프로비저닝 | NS + RBAC + PV/PVC + Secret + Gateway + VirtualService 등을 한번에 생성. k8s-cicd-init과 k8s-app-init 두 저장소에 동시 작업. 기존 5개 슬라이드(목록/생성/상세/환경추가/삭제) 재활용 |
| **네임스페이스 리소스** | 단일 리소스 관리 | K8s namespace를 직접 생성/조회/삭제하는 kubectl 대체 화면. 신규 기획 |

### 네이밍 결정 근거

- "워크스페이스": 실제로 namespace 하나가 아니라 **작업에 필요한 모든 리소스가 세트로 생성**되므로, "작업 공간"이라는 의미가 정확히 부합
- "네임스페이스 리소스": K8s 네이티브 용어 그대로 사용하되, 대메뉴 "네임스페이스"와의 이름 충돌을 피하기 위해 "리소스" 접미어 추가

## 3. 프로비저닝 상세 (워크스페이스 생성 시 일어나는 일)

### k8s-cicd-init 저장소

`_template-cicd/` 폴더를 복사하여 `{name}-cicd/` 생성 후 아래 리소스를 배포:

| 리소스 | 설명 |
|--------|------|
| Namespace | `{name}-cicd` |
| ServiceAccount | `cicdbot` (admin ClusterRole 바인딩) |
| LimitRange | Tekton 워커 기본 리소스 제한 (CPU 1, Mem 2Gi) |
| LocalQueue | `tekton-worker-lq` → ClusterQueue 연결 |
| PV x2 | local (`/CICD-DATA/local/{name}-cicd`) + store (`/CICD-DATA/store/{name}-cicd`) |
| PVC x2 | `cicdvol-shared-local-pvc`, `cicdvol-shared-store-pvc` |
| Secret 다수 | Git, Docker, Nexus, Kubeconfig, SSH, CA, TLS, ArgoCD |
| RoleBinding | Tekton Dashboard tenant-owners / tenant-runners |

### k8s-app-init 저장소

각 환경(dev/stg/prd)별 `sample-{env}` 폴더를 복사하여 `{name}-{env}/` 생성:

| 리소스 | 설명 |
|--------|------|
| Namespace | `{name}-{env}` |
| Secret | `regcred` (Docker pull) + `secret-tls` (TLS) |
| Istio Gateway | `shared-gw` (HTTP→HTTPS 리다이렉트, TLS 종료) |
| VirtualService | `{env}-{name}.cone-chain.com` 호스트 라우팅 |
| AuthorizationPolicy | Istio 인증/인가 정책 (dev 환경 전용) |

## 4. 영향 범위

### 슬라이드 변경

| 대상 | 변경 내용 |
|------|----------|
| 기존 5개 슬라이드 (CCP-NS-001~007) | 폴더 위치 이동, GNB preset 변경 (cicd → namespace) |
| 사이드 메뉴 설정 (`sideMenuConfig.tsx`) | cicdSections에서 네임스페이스 제거, namespace에 sections 추가 |
| GNB 프리셋 (`gnb-presets.tsx`) | namespace용 프리셋 추가 |
| 신규 슬라이드 | 네임스페이스 리소스 목록/생성/삭제 화면 |

### CI/CD 메뉴 변경

```
CI/CD (변경 후):
├─ 저장소 (StorageClasses, PV, PVC, ConfigMaps, Secrets)
├─ 파이프라인
└─ 카탈로그
```

네임스페이스가 빠지면서 CI/CD는 파이프라인/저장소/카탈로그에 집중.

---

## 5. 기대 효과

- 메뉴 중복 해소 및 정보 구조 정리
- "워크스페이스 = 작업 환경 세트" vs "네임스페이스 리소스 = 단일 K8s 리소스" 개념 분리 명확화
- CI/CD 메뉴의 역할 집중 (파이프라인 실행/관리)
- kubectl 없이 웹에서 namespace 직접 관리 가능
